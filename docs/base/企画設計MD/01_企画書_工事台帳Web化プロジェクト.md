## 1. プロジェクト概要

### 1.1 背景

株式会社クラップ（福井県坂井市）では、現在Excelで運用している工事台帳一式（工事台帳本体／QCDS原価算定表／見積条件書／見積書表紙／内訳書／注文書・請書／請求書／印紙税額算定表）を、社内6〜20名で同時利用する業務基盤として運用している。年間案件数は50〜200件規模。

Excel運用の課題：

- **同時編集ができない**：複数名が同じファイルを開くと排他制御で誰かが閉じるまで編集できない
- **集計と可視化ができない**：全案件横断の損益、進捗、業者別単価などが見えない
- **データの再利用ができない**：過去業者見積の単価、類似工事の構成、業者マスタなどがファイルに閉じている
- **業者見積の取り込みが手作業**：PDF・紙・Excelで来る業者見積を担当者が手入力で台帳・QCDSに転記している
- **権限と監査がない**：誰がいつ何を変更したかが追えない
- **属人化**：Excelに埋め込まれた計算式の意味が一部の担当者しかわからない

### 1.2 過去の試み

`G:\マイドライブ\antigravity\Construction_Manager_v2` として一度開発を行ったが、完成度が不十分だったため、本企画でゼロから再構築する。

### 1.3 目的

1. Excel台帳と同等以上の入力体験をWebブラウザで提供し、社内のどこからでも同時編集できるようにする
2. 業者見積（PDF・画像・Excel）のスキャン解析を自動化し、台帳・QCDS・見積書への転記工数を削減する
3. 案件横断の集計・グラフ・ステータスを誰でも一目で把握できるダッシュボードを提供する
4. 編集履歴・権限・写真管理など、Excelでは実現できない管理機能を追加する
5. 既存のExcelワークフローを廃止せず、Excel派の従業員も並行利用できる移行パスを担保する

### 1.4 非目標（本プロジェクトでやらないこと）

- 会計システムとの直接連携（将来検討）
- 工程管理ツール（ガントチャート専用機能、工程表自動生成）の作り込み
- モバイル専用ネイティブアプリ（PWAでスマホ対応はする）
- 給与・人事・勤怠管理機能

## 3. 技術選定

### 3.1 確定事項（前回開発と環境を踏まえて）

| 区分 | 選定 | 理由 |
|---|---|---|
| デプロイ環境 | WebARENA Indigo VPS（Ubuntu 22.04、2vCPU/4GB、東京リージョン） | 既存環境を活用、Coolifyで管理 |
| 管理レイヤー | Coolify | 既存運用と統一 |
| バックエンド | Python 3.11 + FastAPI | 既存invoice appと統一、AIライブラリ充実 |
| フロントエンド | Next.js 14（App Router）+ TypeScript | 既存LLMO SaaSと統一、SSR/SSG柔軟 |
| UIライブラリ | shadcn/ui + Tailwind CSS | デザイン統一、コンポーネント豊富 |
| データベース | PostgreSQL 16（自己ホスト、Coolify上） | invoice appと同居可、移行コスト低 |
| ORM | SQLAlchemy 2.0 + Alembic | 既存FastAPIスタックと統一 |
| 認証 | FastAPI Users + JWT（refresh token） | 既存LLMO SaaSと統一 |
| AI（見積解析） | Google Gemini API（gemini-2.5-pro Vision） | コスト・精度・PDF/画像対応の総合判断 |
| ファイルストレージ | Phase 1: VPSローカル / Phase 2: Canonサーバー（SMB同期） | 段階的に統合 |
| 帳票生成（PDF） | WeasyPrint（HTML→PDF） | レイアウト忠実度が高い、CSS制御可能 |
| 帳票生成（Excel） | openpyxl（既存テンプレートに値埋め） | レイアウト完全維持 |
| グラフ可視化 | Recharts | Next.jsと相性良し、レスポンシブ |
| ホスティングドメイン | 例：`koujidaichou.clap-corp.example`（要MuuMuu DNS設定） | サブドメイン運用 |

### 3.2 Geminiモデル選定の根拠

業者見積は「PDF・画像・Excel／バラバラのレイアウト」のため、Vision対応で表構造の読み取り精度が高いモデルが必要。`gemini-2.5-pro` は表認識性能が `gemini-2.5-flash` より高く、業者見積のように構造化が必須な用途に向く。バッチ処理は `flash` で十分な場合もあるため、コスト最適化として「初回スキャン＝pro、簡易再認識＝flash」のフォールバック構成にする。

## 4. スコープとフェーズ分割

### Phase 1（コア機能、リリース必須）

期間：4〜6週間（Claude Code活用前提）

- 認証・権限・編集履歴の基盤
- 工事台帳CRUD、QCDS、ステータス管理
- 見積書／注文書／請求書のPDF＋Excel出力
- 業者見積スキャン（Gemini API）→ QCDS／取決見通表／見積内訳への転記
- Excelインポート（既存Excel台帳の取り込み）
- 業者マスタと過去単価履歴
- 写真・図面のVPSローカル保存
- ダッシュボード（全案件集計＋案件カードビュー）
- 印紙税自動算定

### Phase 2（拡張、リリース後3ヶ月以内）

- Canonサーバー連携（SMB経由で写真・図面同期）
- Slack/メール通知
- 高度な検索・絞り込み
- 全文検索（PostgreSQL pg_bigm）
- カレンダービュー、ガントチャート風工程表
- 外部システム連携用REST API公開

### Phase 3（将来）

- 会計ソフト連携（freee / マネーフォワード）
- スマホアプリ化（PWA→ネイティブ）
- AI見積アシスタント（過去データから新規見積を提案）

## 7. リスクと対策

| リスク | 影響 | 対策 |
|---|---|---|
| Gemini APIの解析精度が業者見積の品質によりバラつく | 取込ミス、手戻り | 解析結果は必ず人がレビューする画面を挟む。低信頼度フィールドは黄色強調 |
| Excelテンプレートの細かなレイアウト崩れ | 顧客提出書類で問題 | openpyxlで既存テンプレートに値埋めのみ。スタイルは触らない |
| Canonサーバーの仕様が不明 | Phase 2が止まる | Phase 1ではVPSローカル＋手動エクスポートで凌ぐ。仕様調査をPhase 1中に着手 |
| 高齢従業員の移行抵抗 | 並行運用で混乱 | Excel往復ができることで安心感を担保。研修と簡易マニュアル整備 |
| 同時編集時の衝突 | データ破損 | 楽観的ロック（updated_atベース）＋編集中表示 |
| 印紙税テーブルの変更 | 法令遵守違反 | テーブルをDBで持ち、管理画面から更新可能にする |
| Geminiの料金変動・モデル廃止 | コスト増・改修 | プロバイダ抽象化レイヤを噛ませ、Claude/GPTにも切り替え可能に |

## 8. 成功基準（KPI）

リリース3ヶ月後に以下を達成：

- Excel手作業に対し、業者見積→台帳転記の所要時間を50%以上削減
- 社内6名以上が日常的に利用（週3回以上アクセス）
- 案件ステータスの「未更新」滞留が30%以下（=現場が更新するようになった）
- 月次の全案件損益が社長画面で見れる状態
- Excelとの並行運用で混乱なく業務継続
