## 0. なぜこの追補が独立しているか（Claude Codeへの重要な注記）

前回開発（Construction_Manager_v2）で本人がClaude Codeに伝えるのに最も手こずった機能。
理由は「単なる承認ワークフロー」ではなく、**承認結果が帳票（PDF/Excel）の見た目に印影として反映される** という二段構えになっているため。

一般的な承認ワークフロー機能は「承認しました」というステータスを記録するだけだが、本機能では：

1. **業務フロー側**：承認依頼 → 通知 → 承認 → ステータス管理
2. **帳票出力側**：承認結果に応じて見積書PDF/Excelの「承認」「審査」「担当」枠に印影を自動配置

の両方を実装する必要がある。Claude Codeはこの二段構えを最初に明確に理解してから実装を始めること。

## 1. 業務フロー全体図

```
[作成者] 見積書をdraftで作成・編集
   │
   ▼
[作成者] 「承認依頼」ボタン押下
   │   ・承認者(担当・審査・承認の3名)を指定 ※審査は省略可
   │   ・コメントを任意で添付
   ▼
[システム] approval_requests レコード作成
   │   ・status = pending
   │   ・指定された承認者にSlack/メール通知
   │   ・見積書のstatusを「承認待ち」に
   ▼
[承認者A:担当] ダッシュボードの「承認待ち」一覧から見積書を開く
   │   ・内容確認 → [承認] or [差戻し]
   ▼
[システム] approval_actions レコード作成
   │   ・status = approved/rejected
   │   ・タイムスタンプ、承認者ID、コメント
   │   ・全員承認完了したら見積書status = approved
   │   ・誰か1人でも差戻ししたら見積書status = rejected、作成者に通知
   ▼
[承認者B:審査] 同様に承認
   ▼
[承認者C:承認(社長/部長)] 同様に承認
   ▼
[システム] 見積書status = approved、全員のデジタル印影が「捺印済」フラグに
   │   ・帳票出力時に印影が自動配置される
   ▼
[作成者] PDF/Excel出力 → 顧客提出
   │
   │ ※もし承認後に編集が必要になった場合
   │
   ▼
[作成者] 見積書を編集（数量変更など）
   ▼
[システム] 編集を検知 → 全承認を自動リセット
   │   ・approval_requests を「revoked」に変更
   │   ・見積書status = draft
   │   ・編集履歴に「承認リセット理由：見積内容が変更されました」
   │   ・関係者全員(過去の承認者・作成者)にSlack/メール通知
   ▼
[作成者] 再度「承認依頼」（最初に戻る）
```

## 5. 編集検知と承認リセット（最重要ロジック）

### 5.1 スナップショットハッシュ方式

「編集すると承認がリセットされる」を確実に動作させるため、**承認依頼時に見積書の内容をハッシュ化して保存** し、保存時に毎回比較する。

```python
def calculate_quote_snapshot_hash(quote: Quote) -> str:
    """見積書の承認対象となる内容をハッシュ化"""
    snapshot = {
        "project_name": quote.project_name_snapshot,
        "issue_date": quote.issue_date.isoformat(),
        "subtotal": str(quote.subtotal),
        "tax_amount": str(quote.tax_amount),
        "total_amount": str(quote.total_amount),
        "items": [
            {
                "row_no": item.row_no,
                "item_name": item.item_name,
                "spec": item.spec,
                "unit": item.unit,
                "quantity": str(item.quantity) if item.quantity else None,
                "unit_price": str(item.unit_price) if item.unit_price else None,
                "amount": str(item.amount) if item.amount else None,
            }
            for item in sorted(quote.items, key=lambda x: x.row_no)
        ],
        "conditions_text": quote.conditions_text,
        "payment_condition": quote.payment_condition,
        "validity_days": quote.validity_days,
        "remarks": quote.remarks,
    }
    canonical = json.dumps(snapshot, sort_keys=True, ensure_ascii=False)
    return hashlib.sha256(canonical.encode("utf-8")).hexdigest()
```

### 5.2 編集検知のタイミング

`PATCH /api/v1/projects/{id}/quotes/{quote_id}` の処理内で：

```python
async def update_quote(quote_id: UUID, payload: QuoteUpdate, db: AsyncSession):
    quote = await get_quote(quote_id, db)

    # 既に承認進行中 or 承認済み なら、編集を検知して承認リセット
    if quote.approval_status in ("pending_approval", "approved"):
        await revoke_approval_request(
            quote.current_approval_request_id,
            reason="見積書が編集されました",
            db=db,
        )
        # 編集後、承認ステータスはdraftに戻る
        quote.approval_status = "draft"

    # 通常の更新処理
    apply_changes(quote, payload)
    await db.commit()
    return quote
```

### 5.3 承認進行中の編集ロック方針

要件として「承認後に編集すると自動的にリセット」を選択しているので、**編集自体は許可するが、リセットが発生する旨を事前に警告する** UIにする。

フロントエンド側：
- 承認待ち or 承認済み の見積書を編集モードにしようとしたとき、確認ダイアログ表示
- 「この見積書は承認済みです。編集すると承認がリセットされます。続行しますか？」
- OKを押した場合のみ編集モードへ遷移

これで、誤操作で承認が吹き飛ぶ事態を防ぎつつ、必要な編集はちゃんとできる。

### 5.4 リセット時の通知

```python
async def revoke_approval_request(
    request_id: UUID,
    reason: str,
    db: AsyncSession,
):
    request = await get_approval_request(request_id, db)
    request.status = "revoked"
    request.revoked_reason = reason
    request.completed_at = datetime.now(timezone.utc)

    # 関係者全員に通知（過去の承認者 + 作成者）
    notify_users = {request.requested_by} | {
        step.approver_id for step in request.steps if step.status == "approved"
    }
    for user_id in notify_users:
        await send_notification(
            user_id=user_id,
            template="approval_revoked",
            context={
                "quote_no": request.entity.quote_number,
                "project_name": request.entity.project.project_name,
                "reason": reason,
            },
        )
    await db.commit()
```

## 6. 帳票出力時の印影配置（最重要：見た目の反映）

### 6.1 Excel出力時の印影貼り付け

既存テンプレート `quote_cover.xlsx` の「承認」「審査」「担当」枠に画像を貼り付ける。
スクショから読み取った座標は概ね以下のセル付近：

| 枠 | 対応セル | 備考 |
|---|---|---|
| 承認 | L15付近 | テンプレートのセル位置を実機確認後に確定 |
| 審査 | P15付近 | |
| 担当 | T15付近 | |

```python
from openpyxl.drawing.image import Image as XLImage
from io import BytesIO

def attach_stamps_to_excel(
    wb: Workbook,
    sheet_name: str,
    approval_request: ApprovalRequest,
):
    ws = wb[sheet_name]

    # ステップごとに対応セルを定義（cell_mappings.yamlで外部化）
    stamp_cells = {
        "担当": "T15",
        "審査": "P15",
        "承認": "L15",
    }

    for step in approval_request.steps:
        if step.status != "approved":
            continue
        cell = stamp_cells.get(step.step_label)
        if not cell:
            continue

        # 承認者の印影PNGをロード
        stamp_path = step.approver.stamp.stamp_image_path
        with open(stamp_path, "rb") as f:
            img_data = BytesIO(f.read())
        img = XLImage(img_data)
        img.width = 60   # px
        img.height = 60  # px
        img.anchor = cell
        ws.add_image(img)
```

### 6.3 「未承認状態でPDF出力するとどうなるか」のルール

| 見積書の状態 | PDF/Excel出力可否 | 印影 |
|---|---|---|
| `draft`（下書き） | 出力可（社内確認用） | 印影なし、ウォーターマーク「DRAFT」表示 |
| `pending_approval`（承認待ち） | 出力可（経過確認用） | 部分的に印影あり、ウォーターマーク「承認待ち」 |
| `approved`（承認済み） | 出力可（顧客提出可） | 全印影あり、ウォーターマークなし |
| `rejected`（差戻し） | 出力可（参考） | 印影なし、ウォーターマーク「差戻し」 |

`approved` 状態でなければ顧客に提出してはいけないことを社内ルールとして明示。

## 7. 印影画像生成サービス

`backend/app/services/stamp_generator.py` を新設。

```python
from PIL import Image, ImageDraw, ImageFont
from pathlib import Path

class StampGenerator:
    """ユーザー名から赤丸印影PNGを動的生成"""

    DEFAULT_SIZE = 200  # 出力サイズ（縦書きで縮小しても綺麗に）
    DEFAULT_COLOR = "#C00000"

    def generate(
        self,
        text: str,
        style: str = "circle_red",
        output_path: Path = None,
    ) -> Path:
        """印影PNG生成"""
        size = self.DEFAULT_SIZE
        img = Image.new("RGBA", (size, size), (255, 255, 255, 0))
        draw = ImageDraw.Draw(img)

        color = self.DEFAULT_COLOR

        if style == "circle_red":
            # 円を描画
            margin = 8
            draw.ellipse(
                (margin, margin, size - margin, size - margin),
                outline=color,
                width=6,
            )
        elif style == "square_red":
            margin = 8
            draw.rectangle(
                (margin, margin, size - margin, size - margin),
                outline=color,
                width=6,
            )

        # 縦書きでテキスト配置
        font_path = "/usr/share/fonts/opentype/noto/NotoSansCJK-Bold.ttc"
        # 文字数に応じてフォントサイズ調整
        char_count = len(text)
        font_size = {1: 90, 2: 70, 3: 50, 4: 40}.get(char_count, 35)
        font = ImageFont.truetype(font_path, font_size)

        # 縦書き：各文字を縦に配置
        total_height = font_size * char_count
        start_y = (size - total_height) // 2
        for i, char in enumerate(text):
            bbox = draw.textbbox((0, 0), char, font=font)
            char_width = bbox[2] - bbox[0]
            x = (size - char_width) // 2
            y = start_y + (i * font_size)
            draw.text((x, y), char, fill=color, font=font)

        if output_path:
            img.save(output_path, "PNG")
            return output_path
        return img
```

ユーザーの印影設定変更時に毎回再生成し、`/var/lib/cmv3/stamps/{user_id}.png` に保存。

| 承認依頼起動 | 各承認者 | 「○○案件の見積書の承認依頼が届きました」+ URL |
| 全員承認完了 | 依頼者 | 「○○案件の見積書が承認されました。顧客提出可能です」 |
| 誰かが差戻し | 依頼者 + 既承認者 | 「○○が差戻しました。コメント: ××」 |
| 編集により承認リセット | 既承認者全員 + 依頼者 | 「○○案件の見積書が編集されたため、承認がリセットされました」 |

### 8.2 メール送信

既存のFastAPIアプリ用に `fastapi-mail` を使用、SMTP経由。
Phase 2で SendGrid に切替する可能性あり（LLMO SaaSと合わせる場合）。

メールテンプレートは `backend/app/templates/email/` にHTML+テキスト両方用意：
- `approval_request.html.j2`
- `approval_completed.html.j2`
- `approval_rejected.html.j2`
- `approval_revoked.html.j2`

## 12. リスクと対策（この機能特有）

| リスク | 対策 |
|---|---|
| 印影が法的に押印と認められるか | 社内承認用途のため、法的効力ではなく業務効率化が目的。顧客提出物は最終的に紙印刷＋手書きサイン or 実印鑑 |
| 承認者が長期不在で業務が止まる | 代理承認機能（管理者が代理で承認できる）を将来追加。Phase 1では未対応 |
| 編集による承認リセット時、関係者が気付かない | メール＋画面上のバッジ通知。さらに編集者本人にも「承認をリセットしました」を明示表示 |
| スナップショットハッシュ計算の処理コスト | 見積書1件あたりPATCH時のみ実行、items含めても数msなので問題なし |
| 印影画像のキャッシュ管理 | ユーザーが印影設定を変更したら即座に再生成。帳票出力時は毎回最新版を読む |
| 帳票テンプレートのセル位置がズレる | `cell_mappings.yaml` で外部化、テンプレ更新時に座標再計測 |

## 13. テスト計画（この機能特有）

### 単体テスト

- スナップショットハッシュの一意性、再現性（同じ内容なら同じハッシュ）
- 印影生成（1〜4文字、各スタイル）
- 承認ステップの順序処理（順次／並列）
- 編集検知ロジック（PATCH時のみ）

### 統合テスト

- 承認依頼→3名承認→PDF出力で印影3つ確認
- 1名承認後に差戻し→ステータス確認、通知確認
- 全員承認後に見積編集→リセット、再依頼→再承認のフロー
- 同時に2人が承認しようとした場合のロック動作

### E2E

- ログイン→見積作成→承認依頼→別アカウントで承認→PDFダウンロード→印影確認
