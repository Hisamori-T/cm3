## 1. プロジェクト概要

**クライアント**：株式会社クラップ（CLAP CORPORATION）
- 福井県坂井市の建設会社
- 社員数：6〜20名規模
- 年間工事案件数：50〜200件

**移行対象**：
30年近く運用されてきたExcelベースの工事台帳一式（工事台帳／QCDS原価算定表／見積書／注文書／請求書）を、Webアプリケーションに移行する。

**重要な前提**：
これは「Excelの完全代替」ではなく「Excelに違和感を持たない人にも受け入れられるWeb版」を目指す。高齢従業員も使うため、見慣れたExcel的なレイアウトと、モダンなWeb UIのバランスが必要。

## 2. デザインの大方針

### 2.1 トーン＆ボイス

| 軸 | 方向性 |
|---|---|
| 雰囲気 | 業務的・実直・信頼感（建設業向け） |
| 派手さ | 控えめ。マーケティング要素ゼロ |
| 文字密度 | 高め（業務情報が大量にある） |
| インタラクション | キーボード操作が快適。Excel経験者が違和感ないTab移動 |
| 余白 | Web標準より少し詰める。Excelに慣れた人が「スカスカ」と感じない密度 |

### 2.2 やってはいけないこと

- **派手なグラデーション、アニメーション、装飾**
- **絵文字の多用**（業務アプリなので不要、ステータスバッジ程度はOK）
- **「AI感」のあるイラスト・キャラクター**
- **過度に丸い角丸**（rounded-2xl以上は使わない）
- **モダンSaaSにありがちな「Hero」「Features」レイアウト** → 業務アプリなので不要
- **Excelの完全模倣** → 罫線まみれの「ただのExcelっぽいHTML」にしない

### 2.3 参考にすべきトーン

雰囲気としては以下の系統が近い：
- Linear（密度高め、業務的、洗練）
- Notion（情報の階層が明確）
- freee（日本の業務アプリとして信頼感）
- マネーフォワード クラウド会計

逆に**避けるべき**：
- Notion AI のようなふわっとしたAIプロダクト感
- Vercel/Stripe トップページのような派手さ
- Material Design 3 の過度に角丸＋カラフルな感じ

## 3. デザインシステム指定

### 3.1 カラーパレット

クラップのロゴは「濃紺（ネイビー）」基調。これをベースカラーに：

| 用途 | カラー | HEX |
|---|---|---|
| Primary（メイン） | クラップネイビー | `#1B2A52` |
| Primary Hover | やや明るく | `#2A3A6B` |
| Accent（重要操作） | 赤系（捺印・警告と区別） | `#0E7C7B` （ティール系推奨） |
| 背景 | ニュートラル白 | `#FAFAF9` |
| カード背景 | 純白 | `#FFFFFF` |
| 境界線 | グレー | `#E5E7EB` |
| テキスト主 | ダークグレー | `#111827` |
| テキスト副 | ミディアムグレー | `#6B7280` |
| 印影・差戻し | 赤 | `#C00000`（印影専用、他のUIには使わない） |

ステータスカラー（7段階）：

| ステータス | カラー | 用途 |
|---|---|---|
| 見積中 | グレー `#9CA3AF` | 着手前 |
| 受注 | ブルー `#3B82F6` | 確定 |
| 着工 | シアン `#06B6D4` | 開始 |
| 施工中 | アンバー `#F59E0B` | 進行中 |
| 完工 | グリーン `#10B981` | 完了 |
| 請求済 | パープル `#8B5CF6` | 請求 |
| 入金済 | エメラルド `#059669` | クローズ |

### 3.2 タイポグラフィ

- **本文**：Inter または system-ui + Noto Sans JP
- **数値（金額・数量）**：等幅フォント（JetBrains Mono / Roboto Mono）。桁揃えが効くこと
- **見出し**：本文と同じファミリー、ウエイトで差別化
- **サイズ階層**：xs(11) / sm(13) / base(14) / lg(16) / xl(18) / 2xl(22) / 3xl(28)

業務アプリなので**base=14pxが基本**。Web一般の16pxより少し小さく、情報密度を稼ぐ。

### 3.3 スペーシング

Tailwindの4pxグリッド準拠だが、テーブル系画面では2pxの細かい刻みも許可：

- カードのパディング：`p-4`（16px）
- セクション間：`gap-6`（24px）
- テーブルセル：`px-3 py-2`（12px × 8px）

### 3.4 コンポーネントライブラリ

**shadcn/ui ベース**で構築する。実装側でshadcn/uiを採用する前提なので、Claude Designの出力もshadcn/uiコンポーネントの組み合わせで構成してください：

- Button, Input, Select, Checkbox, RadioGroup
- Table（密度高め、ストライプなし）
- Dialog, Sheet（モーダル系）
- DropdownMenu, Popover
- Card
- Tabs
- Toast
- Badge（ステータス表示）
- Avatar（印影プレビューもこの枠で）

グラフは **Recharts**：
- LineChart, BarChart, PieChart, ComposedChart
- レスポンシブ対応
- 配色は3.1のステータスカラーと統一

### 3.5 アイコン

**Lucide React** を使用。線細め（stroke-width=1.5）に統一。
- ナビゲーション、アクション、ステータスに使用
- 装飾としては使わない

## 4. 重要な画面とその要件

設計書 Part1 の「4.1 画面一覧」16画面のうち、特に注力してほしい画面を優先度付きで列挙します。

### 優先度A：このプロジェクトの核となる画面

#### S02 ダッシュボード（`/`）

**目的**：社長・部長が「今、会社の工事案件全体がどうなっているか」を5秒で把握できる画面。

**構成要素**：
1. **ヘッダー**：年度切替（2024/2025/2026...）、「自分担当のみ」トグル
2. **KPIカード4枚**（横並び）：
   - 今期受注額：¥45,800,000（前年比 +12%）
   - 今期粗利：¥9,200,000（粗利率 20.1%）
   - 進行中案件数：23件
   - 平均営業利益率：11.3%
3. **ステータス分布**（円グラフ）：7段階の案件数
4. **月別推移**（棒+線の複合グラフ）：受注額（棒）と粗利（線）
5. **利益率ランキング Top10**（テーブル）：工事名、発注者、金額、粗利率、ステータス
6. **期限アラート**（赤帯のカード）：工期超過2件、請求書未発行3件、入金未確認1件
7. **最近の活動**（タイムライン）：誰が・いつ・何の案件を更新したか直近20件

**重要**：6〜20名の組織なので、ダッシュボードはシンプルにして「数字がちゃんと見える」ことを優先。豪華なデータ可視化は不要。

#### S04 案件詳細（工事台帳）（`/projects/[id]`）

**この画面が最重要**。現行Excel「工事台帳」シート（添付スクショ参照）の情報をWebに移したもの。

**Excel現行レイアウト**（添付スクショ）：
- 左上：受注伺（案件）兼 工事台帳のヘッダ
- 中央：工事番号、工事名、工事場所、発注者、工期（見積/契約/実施）、発受注区分、支払条件、工事概要
- 右上：工事割出テーブル（実行予算／取決見通／精算見通の3列、現場経費の内訳）
- 中央下：取決見通表（業者ごとの No / 支払先 / 工種 / 実行予算 / 取決金額 / チェック / 月別支払 / 支払計 / 残支払 が30行）
- 左下：前施工区分、客先担当、当社担当

**Web版での要件**：
- レイアウトはExcelに似せつつ、モダンに整理する
- 上部に「工事番号 / 案件名 / ステータスバッジ」を常時表示
- ステータスドロップダウンで7段階を切り替え可能
- 各セクションをカード状に分離（情報の塊が見える化）
- 取決見通表は横スクロール対応の大きなテーブル（30行 × 月別12列以上）
- セルクリックでインライン編集（Excelっぽい操作感）
- Tabキー移動で隣のセルに移れること

**画面構造案**：
```
┌─ ヘッダー（固定）──────────────────────────┐
│ ロゴ | 工事番号 26-1-001 | 案件名 | [ステータス▼] [編集] [PDF]
├─────────────────────────────────────────┤
│ ┌─ 案件情報（左2/3）──┐ ┌─工事割出（右1/3）─┐ │
│ │ 工事名             │ │ 実行予算/取決/精算  │ │
│ │ 工事場所           │ │ 直接工事費 ...    │ │
│ │ 発注者             │ │ 現場経費 ...      │ │
│ │ 工期(見積/契約/実施)│ │ 経費 ...          │ │
│ │ 区分・支払条件     │ │ 営業利益 ...      │ │
│ │ 工事概要           │ │ 目標営業利益 ...  │ │
│ └─────────────────┘ └─────────────────┘ │
├─────────────────────────────────────────┤
│ ┌─ 取決見通表 / 専門業者取決伺 ────────────┐ │
│ │ No 支払先 工種 実行予算 取決金額 ✓ 差額  │ │
│ │ ... 30行 + 月別支払スクロールテーブル ... │ │

├─────────────────────────────────────────┤
│ ┌─ 担当者情報 ────────┐ ┌─ クイックリンク ─┐ │
```

#### S05 QCDS（`/projects/[id]/qcds`）

QCDS原価算定表。添付の「現行Excelスクショ_QCDS_1.png」「QCDS_2.png」のレイアウトを参考に。

**特徴**：
- A 直接工事（26〜30行の業者・資材リスト）
- B 経費関係（労災保険・工事保険・契約印紙代・売上の領収書・特殊保険・事務用品・通信交通・雑費）
- 事業部経費（現場担当者給与・工事部経費・共通経費・その他）
- C その他経費（一般管理費・金利）
- 計算式が複雑（小計・比率・営業利益①②）
- 入出金予定表（月別現金フロー）

**Web版での工夫**：
- セクションごとに折りたたみ可能なカード
- 数値はリアルタイムで再計算（変更すると関連セルが即座に更新）
- 計算式が入っているセルは編集不可（表示色で区別）
- 比率は%表記でわかりやすく
- 印刷時はExcelのレイアウトに近づく

## 6. デザインの一貫性ルール

### 6.1 ステータスバッジ

7段階のステータスは常に同じ見た目で表示する：

```
[●見積中] [●受注] [●着工] [●施工中] [●完工] [●請求済] [●入金済]
```

- 小さい丸印 + テキスト
- 背景色はステータスカラー（3.1参照）の20%透明度
- 文字色はステータスカラー濃いめ

### 6.2 金額表示

- 必ず3桁カンマ区切り
- 通貨記号は「¥」を前置
- 等幅フォントで桁揃え
- マイナスは赤字＋括弧 `(¥1,000)`
- ¥0 は `-` 表示

### 6.3 日付表示

- 標準：`2026/05/13` または `2026年5月13日`
- 相対：`今日`／`昨日`／`3日前`／`先月` も併用可
- 工期：`2026/04/01 〜 2026/06/30`

### 6.4 編集権限の表示

- 編集可能な場合：通常表示、Hover時にうっすら背景色
- 編集不可（権限なし）：薄いグレー、Hover時にツールチップ「編集権限がありません」
- 編集中（楽観ロック）：黄色枠＋「他の人が編集中」アイコン

### 6.5 承認ステータス表示

見積書の承認ステータスバーは以下4状態：

| 状態 | バー色 | 文言 |
|---|---|---|

## 7. レスポンシブ対応

主にPC使用だが、現場で工事担当者がタブレット・スマホで確認するケースあり：

| デバイス | 想定画面 |
|---|---|
| PC（>=1280px） | フル機能、全画面サポート |
| タブレット（768〜1279px） | フル機能、テーブルは横スクロール |
| スマホ（<768px） | ダッシュボード、案件一覧、案件詳細（読取専用）、進捗（写真アップロード）のみ |

スマホ版は完全ネイティブUI化ではなく、PWA + レスポンシブで対応。

## 9. デザインに反映してほしい業務上のニュアンス

これは設計書には書かれていない、現場のニュアンス：

1. **「数字を間違えない」が最優先**
   工事金額の桁を間違えると会社が傾く。金額入力UIは大きく、確認しやすく、誤操作しにくく。

2. **「承認」は重い行為**
   印影が押されるということは、その人が責任を持つということ。承認ボタンは派手にせず、押す前に内容確認を促すデザインに。

3. **「現場の人」と「事務所の人」の感覚差**
   工事担当者は屋外でスマホ片手に進捗を更新したい。事務担当者はPCで腰を据えて入力したい。それぞれに最適化。

4. **「上司に見せる画面」と「自分が作業する画面」**
   ダッシュボードは社長が見る画面なので、細かい数字より一目でわかる傾向。案件詳細は担当者が作業する画面なので、密度高くてOK。

5. **「現行Excel」への愛着**
   30年近くこのExcelで業務を回してきた人たちがいる。Web版を「裏切り」と感じさせないよう、現行レイアウトの良いところは受け継ぐ。

## 10. デザインプロセスの依頼順序

以下の順序でClaude Designに作ってもらいたい：

### Phase 1：デザインシステム構築（最初）

1. クラップロゴを読み込ませてカラーパレット確定
2. shadcn/ui + Tailwind 設定をベースにデザイントークン定義
3. 主要コンポーネント（Button, Input, Table, Card, Badge, Dialog）のスタイル確定

### Phase 2：核となる画面のプロトタイプ

優先度A の3画面：
1. **S02 ダッシュボード** — 雰囲気が決まるので最初に
2. **S04 案件詳細** — 最も複雑、Excelとのバランス確認
3. **S05 QCDS** — 計算が複雑なテーブルUI

これらができれば、残りの画面の方向性が決まる。

### Phase 3：業務効率化系の画面

4. **S12 スキャン結果レビュー** — 左右2ペインの新しい体験
5. **S06 見積書 + 承認フロー** — 印影UIの試作
6. **S11 スキャンアップロード**

### Phase 4：周辺画面

残り全画面を一気に。

### Phase 5：帳票デザイン

PDF/Excel出力時のレイアウトをHTML+CSSで設計。WeasyPrintで印刷した時のレイアウトを意識。
