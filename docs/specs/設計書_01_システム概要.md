## 1. プロジェクト概要

### 1.1 目的

株式会社クラップが Excel で管理していた「受注伺(案件)兼 工事台帳」をWebアプリケーションに移行する。

| 課題（AS-IS） | 解決（TO-BE） |
|---|---|
| Excel の同時編集不可 | ブラウザから複数人同時編集 |
| 業者見積の手動転記ミス | PDF/Excel をAIが読み取り自動入力 |
| 帳票作成に時間がかかる | クリック1回でPDF/Excel出力 |
| 進捗・出面が別ファイル管理 | 案件に紐付け一元管理 |
| 承認フローがメール/口頭 | アプリ内承認ワークフロー |

```
    │ HTTPS
    ▼

    ├── /api/v1/* ──→ [ cmv3-api (FastAPI port 8000) ]
    │                         │
    │                    ┌────┴────┐
    │                    ▼         ▼
    │             [ PostgreSQL ] [ Redis ]
    │                    ▲         │
    │               [ cmv3-worker (Celery) ]
    │                    │
    │               [ Gemini API ]
    │
    └── /* ──────→ [ cmv3-web (Next.js port 3000) ]
```

### 3.2 Docker Compose サービス構成（本番）

```yaml

  cmv3-api:      # FastAPI バックエンド (port 8000)
  cmv3-web:      # Next.js フロントエンド (port 3000)
  cmv3-worker:   # Celery ワーカー (AI解析・非同期処理)
  cmv3-nginx:    # リバースプロキシ (port 8005→443)
  cmv3-redis:    # Redis (port 6379)
  postgres:      # 既存の共有 PostgreSQL (外部ネットワーク接続)

volumes:
  scan_uploads:  # アップロードファイル共有 (api + worker)

networks:
  cmv3-prod-network:  # 内部通信
```

## 4. 認証・認可設計

### 4.1 認証方式

- **JWT アクセストークン**: 有効期限 30分、`Authorization: Bearer {token}` ヘッダー
- **リフレッシュトークン**: 有効期限 7日、LocalStorage に保存
- **パスワードハッシュ**: Argon2id アルゴリズム

### 4.2 ユーザーロール

| ロール | 説明 | 権限範囲 |
|---|---|---|
| `super_admin` | システム管理者 | 全操作・ロール付与 |
| `admin` | 管理者（社長・部長） | 全操作・ユーザー管理 |
| `manager` | 上長（査閲・承認） | 閲覧・一部承認 |
| `accounting` | 経理 | 注文書・請求書確認 |
| `staff` | 現場・営業 | 担当案件の編集 |
| `member` | 一般（後方互換） | staff 相当 |

### 4.3 編集権限ルール

- 案件の編集: `admin/super_admin` または `created_by == 自分`
- 案件の閲覧: 全認証済みユーザー
- 注文書の発行: `admin/super_admin/accounting/manager`
- ユーザー管理: `admin/super_admin`
- `super_admin` 付与: `super_admin` のみ

## 5. 業務フロー

### 5.1 案件ライフサイクル（7ステータス）

```
見積中 (quote)
    │ 受注確定（注文請書受領）
    ▼
受注 (ordered)
    │ 着工
    ▼
着工 (started)
    │ 施工開始
    ▼
施工中 (in_progress)
    │ 完成
    ▼
完工 (completed)
    │ 請求書発行
    ▼
請求済 (billed)
    │ 入金確認
    ▼
入金済 (paid)
```

### 5.2 見積→注文→請求フロー

```
1. 案件作成（工事番号自動採番）
     │
2. 顧客見積書作成
     │ 承認依頼（担当→確認→承認の順）
     │ 承認ワークフロー完了
     │
3. 業者見積スキャン → AI解析 → QCDS/見積版に取込
     │
4. 注文書作成（業者宛）
     │ 発行済みに変更 → 注文請書を自動発行
     │ 先方押印（signed）
     │
5. 請求書作成
     │ 入金記録 → 全額入金で「入金済み」自動遷移
     │
6. 案件を「入金済み」に更新
```

### 5.3 工事番号の採番規則

```
形式: {西暦下2桁}-{社員番号}-{連番3桁}
例:   26-1-001（2026年、社員番号1、第1号）

ロジック:
- 年・社員番号の組み合わせで project_number_sequences テーブル管理
- SELECT FOR UPDATE で排他制御
- 手動上書き可
```

### 5.4 業者見積スキャン → AI解析フロー

```
1. PDF/Excel/画像をアップロード（最大20MB）
2. Celery ワーカーが非同期で処理開始
3. PDFは pdf2image で画像変換（最大10ページ）
4. Gemini 2.5 Flash API に送信（Vision）
5. 構造化レスポンス（業者名・明細行・金額）をパース
6. scan_results + scan_result_items に保存
7. ユーザーが結果を確認・修正
8. 確定後、QCDS または見積版に転記
```

## 6. 工事台帳の手動入力区別ルール

Excel の「受注伺(案件)兼 工事台帳」をWeb移植する際の視覚的区別ルール:

| 種別 | 背景色 | ボーダー | 挙動 |
|---|---|---|---|
| **自動取得（DB連動）** | 白 | なし | 読み取り専用 |
| **手動入力（未入力）** | 薄黄 `#fffbeb` | 点線 | クリックで編集開始 |
| **手動入力（入力済）** | 白 | 実線 primary | 編集可能 |
| **ロック項目** | 薄グレー | なし | 変更禁止 |
